# AI-Scientist 复现最终报告:资源账本、流程改动与可行性评估

> 对象:Sakana AI《Towards end-to-end automation of AI research》(Nature 651, 2026)所述 AI-Scientist-v2 系统。收盘日期:2026-06-12。
> 本报告回答六个问题:花了多少资源;照搬论文方法是否可行;改了哪些流程、为什么;生成的论文是否可用;解决了/没解决哪些问题;按当前模型能力,"AI 自动产论文"这件事的真实边界在哪里。
> 数据来源:运行日志与 progress.log 时间线、token_tracker 账本、journal 树(1810 节点)、67 份 ensemble 评审、31 份 grounding 审计、预注册锦标赛与校准集(PREREGISTRATION.md / BENCHMARK_REPORT.md §13 / sprint scoreboard 终版)。本报告全部数字在发布前经过逐条对源核查。

---

## 0. 一句话结论

**照搬 Nature 论文的流程"能跑通但不能直接用":** 框架在非官方环境下需要约 12 处环境适配,另有 4 个上游级致命缺陷需要处置(2 处修复/中和、2 处流程规避),才能稳定批量生产;产出的论文在 workshop 短文档位上"分数达标、事实不可信"——原始版 grounded 率(声明可溯源占比)中位仅 ~30%,即**六到八成经验声明无日志支撑或与日志矛盾**,必须外挂声明级审计 gate 把它压到 ≥95% 才算"能用"(终版 26 个论文实例中 25 个达标、17 个 100% 干净)。盲评下我方诚实认证版以 84.4%(95% CI [0.756, 0.905])胜官方 showcase 原版。端到端成本中心估计 **~1.4 亿 token、~1140 GPU·小时(预留)**;冲刺批 37.9 小时墙钟消化 20 个 idea、全部成文。该系统当前定位是"**负结果/诊断类 ML 短文的流水线 + 可审计的实验执行器**",不是"自动科学家"。

## 1. 资源账本

### 1.1 Token 消耗(全战役:ideation → 树搜索 → 写作 → 评测)

| 环节 | 计量方式 | 数量 |
|---|---|---|
| 树搜索(code/feedback,gpt-5.5 为主) | 1810 个节点 × 实测锚点 ~67K token/节点(deepseek 满预算 64 节点 ≈ 4.3M 实测) | **~1.2 亿 token**(按每节点 50K–140K 传导,区间 0.9–2.5 亿;gpt-5.5 含 reasoning,偏上限) |
| writeup / 引用 / 单篇评审 / VLM 看图(gpt-4o 为主) | token_tracker.json 硬记账(18 个 run) | gpt-4o prompt 684 万(其中 cached 279 万)+ completion 122 万,另含 gpt-5.5 ~9 万,合计 **≈816 万 token** |
| 评测链(5 票 ensemble × 67 篇次、31 次 grounding 审计、53 轮 reground 复审、90 场锦标赛、40 例×2 家族校准) | 无账本,按调用数 × 平均上下文估算 | **~1000–1500 万 token** |
| ideation(本机,76 个 idea × 3 reflections) | 无账本,估算 | ~100–200 万 token |
| **合计** | | **中心估计 ~1.4 亿 token**(按树搜索每节点区间传导,全距 ~1.1–2.7 亿) |

- 折算单篇:全战役(含失败的放大档)摊到终版 26 个成文论文实例 ≈ **540 万 token/篇**;若按 gpt-4o 牌价(prompt $2.5/M、completion $10/M)、以硬记账段实测的 output 占比(~15%)粗折,约 **$20/篇 量级**(reasoning 段 output 占比无实测,取更高配比则到 ~$30)。AI-Scientist v1 早期 arXiv 预印本宣称 "less than $15 per paper",量级相当;Nature 正式版未报任何算力/成本数字——全文无 GPU 型号、无 GPU-hours、无美元成本,只给 wall-clock "several hours to over 15h"。
- 关键画像:**token ∝ 节点数,不 ∝ 墙钟**。8h 大预算档 17.5h 只烧了 ~250 万 token(143K/h,为理论满速的 ~1%)——"跑得久"≠"烧得多",长 exec.timeout 把无 token 的等待执行区撑大。

### 1.2 服务器资源

| 维度 | 用量 | 备注 |
|---|---|---|
| GPU 预留 | **~1140 GPU·h**(放大档 72h+33h、冲刺档 37.9h,均 8×H20) | **实际利用率极低**:util 0–12%,每卡显存 0.5–2GB/96GB。本池 idea 全为 T0–T1 档(玩具/小真实集),8×H20 过剩约两个数量级;纯 CPU 也能跑(GPU 是软依赖) |
| CPU | 放大档 8 路并行把 128 逻辑核打爆(load 289–307,2.3× 超订);冲刺档 4 lane × workers2 负载健康 | 放大档失败的三因素之一 |
| 墙钟 | 冲刺批 **37.9h**(2026-06-10 10:31 → 06-12 00:23),20/20 idea 全部跑完(14 个 rc=0,5 个触 10h 上限,1 个 rc=1 写作环节崩) | 单 idea 2.2–10h,中位 ~6h(非超时 idea 中位 ~5.3h),均值 6.3h |
| 磁盘 | sprint 4.7GB + phase2 10GB | journal/实验产物/PDF |
| 失败成本 | 放大档(workers8/seed10/exec8h)72h **0 篇成文**;其中 6 棵半树原料被离线抢救链利用,6/6 成文 | 教训:配置越激进,运维风险越前置 |

### 1.3 效率结论

官方预算档(20/12/12/18,seed3,exec 1h)是实测的"甜点":冲刺批 20 棵树全部成文(14 棵原生直出、6 棵经抢救/补链),按 37.9h × 8 卡 = 12.6 卡·天计 ≈ **1.6 篇/卡·天**(只算原生直出口径则 ~1.1)。瓶颈不是 GPU 而是 LLM 往返延迟(小预算档占墙钟 85–95%)与中转 API 限流。**给该系统升级 GPU 没有意义,买 token 吞吐才有意义。**

- 放大 worker 数还有一个隐藏墙钟税:框架给每个并行 worker 建独立工作目录、数据集无共享缓存,放大配置档实测同一份 CIFAR-10 被 4 个 worker 各下一遍、跨 stage 还会重下(见 §9)——纯 I/O 等待,零 token 零 GPU,把"放大并行度"的边际收益进一步压低。

## 2. 照搬 Nature 论文的方法可行吗

**结论:管线设计可复现、工程实现不可直接用;"开箱即用"在三种意义上都不成立。**

### 2.1 环境意义上:官方隐含了一整套未声明的环境假设

论文/仓库默认:OpenAI 官方 API、Semantic Scholar key、直连 HuggingFace/arXiv、本地 sudo + 完整 LaTeX。任一不满足就需要适配(本项目逐一命中):

- 中转 API → 模型名规范化(`_route_model()`)、白名单扩充、空 content 兜底;
- 无 S2 key → backoff 无限重试卡死(须加 max_tries),最终换 OpenAlex drop-in 实现真引用;
- arXiv 出口不通 → 引用搜索整体改道 OpenAlex;
- 无 sudo → texlive 用户态自装;
- reasoning 模型(gpt-5.5 类)→ ReAct 解析器、code-only 输出、LaTeX fence 提取、max_tokens 4096→16384 等 6 处鲁棒化,否则静默失败。

**这类改动按 patch 条目计 ≈ 12 处(拆到文件修改点约 20 处),定性为"环境适配",不是论文方法的问题,但说明复现成本被严重低估。**

### 2.2 工程意义上:框架自带 4 个会杀掉 run 或毁掉产物的缺陷(任何环境都会撞)

1. `parallel_agent.py` 工作节点异常被捕获后又 `raise` → 单节点失败杀全 run。**已修复**(去掉 raise,记录后跳过坏节点);仅此一处修复就把全流程成功率从 2/5 提到 4/5;
2. 收尾清理用 psutil 按关键词杀**全机** python 进程 → 并行批产时第一个完成者杀光所有同伴。**已中和**(清空关键词列表);
3. writeup 重试会先删旧 reflection PDF + 成功标志位不可靠 → 与并行读 PDF 的工具竞态、毁产物。**流程规避**(一切读 PDF 的工具等 driver 全退出),列入上游 issue 清单;
4. `find_pdf_path_for_review` UnboundLocalError → 树跑完、论文写完,最后一步评审崩掉整个 rc。**未修**,由离线补链兜底,列入上游 issue 清单。

### 2.3 科学意义上:流程能产出"像论文的东西",但官方流程没有事实保障环节

官方管线 writeup 之后直接评审,**没有任何声明↔日志核对**。终版 26 个实例的原始版 grounded 率:中位 ~30%,17 篇在 14–40%,2 篇 ~53%;另有 2 篇抢救树的"原始版"达 95/98%——后者恰是反证:**抢救链的 writeup 输入是从日志重建的真实 summary,输入真实时编造骤降,说明造假率主要由 writeup 的输入质量决定,而非模型偶发**。官方 3 篇 showcase 干净版无公开日志、只能接受更弱的纯文本一致性审计,结果 0/3 通过(grounded 比率 0.35–0.63,内部矛盾率 14–32%)。**照搬官方流程会批量生产"高分但大半声明无支撑的短文"。可行性的真正分水岭在于是否外挂零造假 gate(见 §3.3)。**

## 3. 我们改了哪些流程、为什么

三类共 ~25 处改动(12 环境适配 + 4 缺陷处置 + 9 项方法学增强),动机各不相同:

### 3.1 环境适配(因为我们的环境 ≠ 官方假设)——12 处

见 §2.1。代表:模型路由规范化、OpenAlex 引用、texlive 用户态、reasoning 模型鲁棒化、补装 psutil/scikit-learn。**这些不改,系统在本环境下根本启动不了或静默产废。**

### 3.2 框架缺陷处置(上游缺陷,与环境无关)——4 处

见 §2.2:2 处修复/中和(异常重抛、psutil 全机杀),2 处流程规避兜底(写作重试删 PDF 竞态、评审入口 UnboundLocalError),全部整理为上游 issue 候选清单。**不处置这些,批量生产的成功率和并行度无法保障。**

### 3.3 方法学增强(官方流程没有、我们加的)——核心增量

| 增强 | 为什么加 | 实测效果 |
|---|---|---|
| **零造假 gate**:grounding 审计(声明抽取→证据包→逐条判定)→ 手术式定点改写(find/replace,≤3 轮)→ 重编译重审 | 原始论文 grounded 中位 ~30%,直接用 = 学术不端 | 26 个实例走完 gate 链:25/26 达 ≥95% grounded、17/26 达 100% CLEAN(1 篇止步 93%,按预注册规则标记、不进 showcase);"诚实税"均值仅 **-0.28** 分,7 篇零税(注:该差值继承绝对尺 0.2 量化噪声底,不可在 band 内精排;"代价小"的举证主权移交给对粗尺稳健的证据——零税篇的精确 0 差值,以及机制证据:输入真实 summary 时 raw grounded 即达 95/98%) |
| **审计器校准**(20 真 + 20 注入伪造,双模型家族盲判) | 审计器自身是 LLM,必须证明它不瞎判 | recall 100%、FPR 5%、跨家族 κ=0.849(注:headline 数来自 in-loop 的 gpt-5.5 审计器,跨家族 deepseek 判定更保守,held-out 终审未做) |
| **5 票异构 ensemble 评审**(gpt-5.5×3 + gpt-4o×2 + meta) | 对齐官方 o4-mini×5 的仪器强度,消单票方差 | 发现绝对尺 mode-collapse:全部机器论文(我方原始/诚实版、官方、旧 4 篇)压缩在 1.8–2.6 |
| **成对盲评锦标赛**(双顺序 × 3 评委家族 × Wilson CI,预注册) | 绝对尺钝刀无法判优越性;官方 6.33 是人类尺,不可比 | 我方 5 篇诚实版 vs 官方 3 篇干净原版:**84.4% [0.756, 0.905]**,逐对 CI 下界全 >50% |
| **预注册协议**(选择规则/声明层级/校准承诺,数据完成前冻结) | 防止"先看结果再定口径"的自我造假 | PREREGISTRATION.md,commit 在数据完成前 |
| **聚合修复**(把 npy 真实嵌套结构 dump 给 LLM 再写聚合脚本) | 官方让 LLM 盲猜数据结构,5 轮 reflection 全失败、图全空 | 一次命中 9 张图;"成功日志 + 空产物"是官方流程的常见静默失败 |
| **离线抢救链**(从 journal 重建 → 补 summary → 聚合 → 写作 → 评审) | 触 10h 上限/写作崩的树原料完整,弃掉浪费 | 被杀的树 6/6 抢救成文,~40min/篇 |
| **诚实重写**(从 grounded 声明集重写全文,重过 gate) | 测试"删假"与"重写"两种诚实化路径的分数差 | 重写版可保住 2.4(零诚实税) |
| **/responses 后端适配器**(priority tier 只在 /v1/responses 生效) | 中转 chat 路径不认 fast 档 | 一棵完整树(idea21)验证零回退;首个金丝雀(idea5)在适配器 bug 期 263 次调用全部安全回退 chat、零伤害,修复后才有零回退 |

**为什么改:不是论文方法错了要改,而是(a)环境不同必须适配,(b)框架工程质量不到批产标准,(c)官方流程缺少事实保障与合法的优越性度量——后者是我们对方法本身的增量。**

### 4.2 "能用"分场景判断

| 场景 | 判断 |
|---|---|
| 作为 workshop 短文(负结果/诊断类)投稿草稿 | **接近能用**:事实层过审计、图表真实、引用真实;但 Presentation 弱(机器论文实测 1.0–2.0,人类校准论文 2.6–3.8——这是机器与人类论文区分度最大的维度,官方 showcase 也只有 1.0–1.6),需人工润色;且 AI 署名伦理须按 venue 政策处理 |
| 作为内部技术报告/实验记录 | **能用**:诚实认证版的每条结论可溯源,比人类随手写的实验记录更可审计 |
| 作为 main conference 论文 | **不能用**:novelty 天花板低(成熟方向小变体)、实验规模 T0–T1、写作深度不足 |
| 不加 gate 直接采信原始输出 | **绝对不能用**:grounded 中位 ~30% 意味着大部分论文含大量无支撑内容 |

## 5. 这轮解决了哪些问题

1. **把"能不能跑"变成"能不能批产"**:4 lane 并行编排 + 进程隔离 + 逐 idea 硬上限 + 抢救链,20/20 idea 全消化,无一漏跑;
2. **把"分数高"变成"分数可信"**:预注册 + 校准过的审计器 + 双仪器(绝对尺只判同档、锦标赛判优越),G1 结论在方法学上站得住;
3. **把"编造不可避免"变成"编造可度量、可修复"**:首次测得原始 grounded 率(中位 ~30%,主体 14–40%)与诚实税(均值 -0.28,7/26 零税)两个量,并发现造假率由 writeup 输入质量主导(输入真实 summary 的抢救版原始 grounded 即达 95/98%)——**证明 gate 的代价很小、收益是质变,这是对官方流程最有价值的增量发现**(代价小这一论断不靠 -0.28 这个绝对尺差值本身:该差值继承绝对尺 0.2 量化噪声底,不可 band 内精排,举证主权落在对粗尺稳健的证据上——零税篇的精确 0 差值,与造假率由输入质量主导的机制证据:输入真实 summary 时 raw grounded 即达 95/98%);
4. **定位 4 个上游级缺陷**:其中 parallel_agent 异常重抛一处修复即把全流程成功率从 2/5 提到 4/5,其余 3 个的处置保障了并行批产不互杀、产物不被竞态毁掉;
5. **算力画像纠偏**:证明瓶颈在 LLM 往返而非 GPU;exec.timeout 的 prompt 引导会制度性地把实验做小——这解释了官方论文为何不报算力。

## 6. 没遇到的、没解决的问题

### 6.1 这轮实验设计上回避了(所以"没遇到"≠"不存在")

- **重负载实验(T2–T3)**:放大档失败后,冲刺池刻意选 1h 量级 idea;DistilBERT/Tiny-ImageNet 级别的 idea 在本框架下仍未跑通过(单节点 timeout 硬墙 + CPU 超订问题未根治);
- **30+ 节点无人区**:官方 Fig.3c 最多测到 ~30 节点,"更多算力是否继续涨分"仍是外推,放大档失败让这个问题保持开放;
- **真实投稿与人类评审**:全部评测是 LLM 评委;LLM 分数与人类接收决策之间的效度鸿沟未触碰(官方 6.33 人类尺与机器尺不可比,我们绕开了而非解决了);
- **数据污染**:MNIST/CIFAR 上的"发现"可能在模型预训练语料里——本轮未做污染检查;
- **idea 查重**:无 S2 key,文献级查重未纳入 ideation 管线;idea 池的"新颖性"只有 LLM 评委共识背书;
- **同 idea 多 attempt 方差**:每个 idea 只跑一次,run 间方差(官方已知很大)未量化。

### 6.2 触发了但只缓解、没根治

- **写作质量天花板**:Presentation 维度机器论文全体落在 1.0–2.0(人类校准 2.6–3.8),诚实重写只能保分不能显著提分;
- **绝对尺 mode-collapse**:5 票 ensemble 把全部机器论文压在 1.8–2.6,分辨率不足以做精细排序(锦标赛是补救而非治愈);
- **审计只能删不能补**:发现"图—文不符"时只能删声明,不能自动补做实验回填证据;
- **框架可观测性**:stage summary 只在全部阶段完成后落盘、成功标志不可靠——靠外挂轮询和事后核对兜底,未改框架本身;
- **引用相关性**:OpenAlex 关键词检索能保真但相关性一般,related work 仍是弱项;
- **无共享数据缓存**:每个并行 worker 用独立工作目录,数据集按 cwd 各下一份、跨 stage 不复用 → 放大 worker 数时数据下载墙钟线性膨胀(放大配置档实测 CIFAR-10 ×4 重下吃掉约 3h 纯 I/O,见 §9)。本轮靠"选无需大下载的 idea"规避,未改框架(可由共享 TORCH_HOME/HF_HOME 或预热软链缓解;根治需框架支持 worker 间共享 data 目录)。

### 6.3 超出本轮范围的开放问题

- 方向选择的品味:系统偏好"安全可跑"的小变体,不会主动选高风险高价值方向;
- 跨论文知识积累:每篇论文独立冷启动,没有"研究纲领"概念;
- 自我改进闭环:系统不会从自己的失败 run 学习(本轮所有教训是人提炼的)。

## 7. 能力边界(按模型/搜索/方向选择三轴)

**模型能力**:gpt-5.5 写实验代码 good 率 ~66%(满预算 63 节点中 42 good),与 gpt-4o(68%)同档,是最强代码大脑之一、且是唯一自发使用真实数据集(MNIST/Fashion/CIFAR)的模型;但 (a) 写作时系统性编造叙事(这是 LLM 的本性,不是 bug),(b) 格式敏感环节(LaTeX/JSON)不如 gpt-4o 稳,(c) 评审时在中低分段 mode-collapse。**"会做实验"与"会诚实报告"是两种能力,后者必须外部强制。**

**搜索能力**:树搜索本质是贪心 + 有限回溯,探索由 prompt 引导而非价值估计;exec.timeout 写进 prompt 后系统性偏向轻实验。文献搜索是关键词级,没有语义级的"这个 idea 在文献里站在哪"的理解。**两层搜索都够用但不智能。**

**方向选择能力**:76 个 ideation 产出里,真正值得跑的 ~40 个是 5 个异构 LLM 评委 + 人工裁决共同筛出的;degenerate idea(效应 < seed 方差)LLM 自己发现不了。**选题是当前人机分工里人占比最大的环节。**

**实验域硬边界**:只能做"代码即实验"的领域。理论舒适区是 T1–T2(小真实集到中等 benchmark),但本轮实测验证的只有 T0–T1;T2(DistilBERT/Tiny-ImageNet 级)未跑通(见 §6.1),T3(7B 微调级)撞单节点 timeout 硬墙。无物理环境、无湿实验、无人类被试。本轮全部产出都在"小数据集 + 小模型 + 训练动力学诊断"这个安全区内——**这不是选择,是当前架构的边界。**

## 8. 总体判断与后续路线

**可行的定位**(以本轮证据):
1. 负结果/诊断类 ML workshop 短文流水线(配 gate,~1.6 篇/卡·天,token 边际成本 ~$20–30/篇 量级);
2. 可审计的自动实验执行器(树搜索 + journal 溯源比人类记录更规范);
3. 作为研究对象本身(框架缺陷、grounded 率、诚实税都是可发表的测量)。

**不可行的定位**(当前):无人值守长程运行(运维事故密度高)、main conference 级产出、无审计直接采信、任何非代码实验领域。

**后续优先级**:
1. 服务器侧 supervisor + 论文状态机账本(根治"靠轮询和人盯"的运维模式);
2. 上游 issue 提交(4 个缺陷 + 复现说明);
3. /responses priority 全量铺开(单树已验证零回退,回退路径经 263 次实战零伤害);
4. 重负载档重试:解 workers 钳制 + 分阶段 timeout,探 T2–T3 与 30+ 节点区;
5. 写作质量专项(在锁死事实集的前提下允许重组织,目标把 Presentation 从 ~1.4 提向人类下限 2.6);
6. 人类评审对照实验(LLM 尺 ↔ 人类尺的效度标定,G1 的最后一块拼图)。

## 9. 2026-06-13 补充:放大配置档试点 + 数据缓存发现

6/12 收盘后,用解钳后的放大配置(每篇 4×H20、num_workers 3、exec.timeout 4h、外层 cap 15h)补跑 2 个 T1 idea,验证"放大并行配置是否提分 + 暴露新瓶颈"。两篇均经完整 gate 链(树搜索 → 写作 → grounding 审计 → reground → 5 票评审),已并入 scoreboard(§4.1 全集 28 实例)。

| lane | idea | 墙钟 | 收尾路径 | grounded raw→终 | raw→诚实(税) |
|---|---|---|---|---|---|
| 0 | training_stats_early_stopping | ~3.5h 端到端 | 树自然完成 → supervisor 自动 writeup+链 | 89.1% → 95.5% | 2.4 → 2.4(零税) |
| 1 | weight_decay_scaling_law | 撞 15h cap 强杀 | supervisor KILLED 路径 → 离线抢救链 | 93.8% → 98.1% | 2.4 → 2.0(−0.4) |

**三个结论:**

1. **放大配置不提分**:两篇 raw 都是 2.4、诚实 2.0–2.4,仍落在全集 1.8–2.4 带内。解 workers 钳制、给 4 卡、放长 exec 没有把分数推出机器论文的 mode-collapse 区间——印证 §6.2「绝对尺 mode-collapse」,绝对分无区分度的结论延伸到放大档。
2. **墙钟方差爆炸**:lane0 3.5h、lane1 撞 15h cap,同档两篇墙钟差约 4×,远大于冲刺档(2.2–10h)的离散度。放大配置把方差源(substage 增殖、数据下载、长 exec 等待)同时放大,使逐 idea 上限(cap)从"兜底"变成"主导墙钟"。
3. **数据缓存是新瓶颈**:lane1 拆账发现框架给每个并行 worker 建独立工作目录,torchvision/HF 数据集的下载 root 相对 cwd,导致**同一份 CIFAR-10 被 4 个 worker 各下载一遍、进入 stage_3 又重下一轮**;实测每份 170MB 走慢中转管道,4 份并发吃掉约 3h 纯 I/O 墙钟(零 token、零 GPU、零进度)。这是 lane0(合成/小数据)3.5h 跑完而 lane1 撞 cap 的部分原因——墙钟差异不全来自 idea 难度。缓解见 §6.2,候选第 7 个上游 issue。

**边界澄清**:这两个 idea 仍是 T1 档(CIFAR-10/100、训练动力学诊断),不是真 T2(DistilBERT/Tiny-ImageNet 级)——本轮只验证了"放大并行配置在 T1 上的行为",§6.1「T2–T3 未跑通」的结论不变。

**对 §8 后续优先级的落地状态**:第 1 项(supervisor 状态机)已上线并经 DONE/KILLED 双路径无人值守实战;第 2 项(上游 issue)6 个已提交;第 3 项(/responses 全量)已铺开,抢救链中再遇中转 500 时自动回退 chat 零中断;第 4 项(重负载档重试)完成放大配置档的 T1 试点,真 T2–T3 与 30+ 节点区仍开放。
